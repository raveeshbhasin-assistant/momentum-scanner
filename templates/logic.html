v3.6.2 Released 2026-04-23 Current
Performance — Avg % / pick metric. The /performance page gains a new KPI card and a third equity-curve toggle alongside R and $ / share. "% / pick" answers: "how much did the typical pick move, as a percentage of its own entry price?" Equal-weighted across picks (Option 1 — not capital-weighted), so a +2% move on a $500 stock and a +2% move on a $20 stock count the same. Complements R (risk-normalized) and $ / share (sizing-dollar view) with a third, price-normalized lens.
Why: "Net $ / share" is easy to misread because higher-priced tickers dominate it in absolute terms — NVDA contributing +$2 is a smaller move than F contributing +$1 in percentage terms, but the old KPI made them look 2:1. "Avg % / pick" is the metric most traders intuitively translate to "is the scanner's typical pick worth trading?" — it's what gets compared to a benchmark like SPY-intraday or a fixed-% stop strategy.
Added
  • pnl_pct on every performance entry. Computed in performance_engine.normalize_entry as (pnl_dollar / entry) × 100, rounded to 3 decimals. Null-safe if either side is missing. The existing JSON store doesn't need re-seeding — derivation happens on every page render.
  • New rollup fields: total_pct, avg_pct, n_pct. Added to _rollup() and its by_outcome sub-split so range / cumulative mode can aggregate % even when per-pick rows aren't present. n_pct tracks how many picks in the group actually carry a valid % (not all do — picks without a resolve price have null pct), so the average divides correctly.
  • day_pct and cum_pct on _daily_series. Mirrors the existing R / $ fields so the equity-curve chart can plot the % series without any extra server work.
  • KPI card "Avg % / pick". Shows the equally-weighted mean of per-pick % across the filtered view. Sub-line shows Σ total% · N picks so you can also see the sum and the denominator. Green if positive, red if negative.
  • Equity curve — third metric toggle. The existing R / $ / share segmented-toggle gains a % button. Works in both single-day mode (per-pick % accumulation) and cumulative/range mode (daily sum of %). Y-axis title and tooltip formatter all branch on the active metric.
  • Picks table — new P&L % column. Sortable like the existing R and P&L columns, rendered via fmtPct. Colour-matches the R cell (positive green, negative red).

"RTH Gate on Re-Entry Dedup + Daily Finds Persistence" — two bugs at a single scheduler seam. The 9:00 ET and 16:30 ET main-scan fires now neither consume a ticker's once-per-day dedup slot nor persist to the daily-finds JSON, so every strong ticker gets its guaranteed one-per-day showing during regular session and /today / /history stay free of pre-market and post-close noise. The in-memory live dashboard still reflects every scan regardless of session, so operators don't lose visibility.
Why: The APScheduler cron for scheduled_scan is hour="9-16", minute="0,30" — it fires at 9:00 and 16:30 ET in addition to the 30 in-session slots. Two independent bugs rode on those two fires. Bug 1: the 9:00 fire was populating _today_tickers_seen, so any ticker that flashed strong on thin pre-market tape (typically un-actionable on Fidelity) silently consumed its once-per-day slot and the legitimate 9:30+ emissions were suppressed as "re-entries." Same root cause at 16:30. This undercut what v3.5.8's Resend notifier could ever send. Bug 2: every fire — in- or out-of-session — was calling add_signals_to_daily(), so data/{date}.json carried pre-market and post-close rows that polluted /today and /history. (The performance log itself was always safe: daily_analysis.filter_market_hours drops anything outside 09:30–16:00 before it hits performance_log.json, which is why 4/20 and 4/22 have zero post-close rows there.) Both bugs live at the same scheduler seam, so they share one fix: compute an RTH flag once per scan and gate both sites on it.

"Market-Hours + Category Filters + Test Hook" — the Resend notifier now only fires during the US equity regular session (Mon–Fri 09:30–16:00 ET) and only when at least one strong signal lands in Category A or B. A new POST /api/notify/test endpoint sends a filter-bypassing round-trip email so Railway env vars can be verified anytime, not just during a live Cat A/B hit.
Why: Two complaints shaped this release. First, the notifier was waking the phone on pre-market and after-hours scans — signals Raveesh can't act on from Fidelity anyway. The fix is two independent gates at the notifier boundary: a clock-based one (NOTIFY_MARKET_HOURS_ONLY, default on) and a quality-based one (NOTIFY_CATEGORIES, default "A,B"). Both use the same category rule the Performance page uses (composite_score ≥ 60 + leadership label in {LEADER, FOLLOWER, LAGGARD, SOLO_MOVER}), so what you see in the email matches what you see on the dashboard. The test-email endpoint was added because the only way to verify a Resend setup used to be waiting for a real Cat A/B signal during market hours — impractical for anyone setting this up on a weekend.
Added

"Range View + Cumulative R Fix" — the Performance page now has a third mode alongside Single day and Cumulative: a Range picker that rolls up any arbitrary start/end window. At the same time, the Total R and Net $ headline cards finally show real numbers in Cumulative mode instead of "—".
Why: Once the log spans multiple days, "all of it" (Cumulative) and "just today" (Single day) stop being enough — you want to slice a week, a strategy-change boundary, or a single past date. Adding a calendar-driven Range picker covers that without building a second page. The R/$ bug was a separate, longstanding quirk: the UI re-computed the headline by re-summing per-pick rows, but in Cumulative mode those rows were synthesized with r_realized: null, so the sum was always zero-against-empty and the card rendered "—". Both fixes share the same engine change (an outcome-split sub-dict on every rollup), so they ship together.
Added
  • Range mode on the Performance page. A new "Range" chip joins "Single day" and "Cumulative" in the date-control group. Activating it swaps the day-dropdown for two <input type="date"> pickers (start / end) bounded to the first and last dates present in performance_log.json. Both inputs default to the full span. Changing either input invalidates the cached response and refetches. The rolled-up KPIs, category cards, time grid, and picks table all re-render against the selected window. The picks table gains a Date column when Range is active so multi-day picks are distinguishable.
  • GET /api/performance/range?start=YYYY-MM-DD&end=YYYY-MM-DD. Read-only projection over the existing log — filters entries to start ≤ date ≤ end and runs the same _bucketed_rollup primitive that powers Single day and Cumulative. Returns the same shape as /api/performance but with range instead of cumulative/single_day, plus days (list of dates actually present in the window) and picks (sorted by date → batch_time → ticker). Validates both params as YYYY-MM-DD and rejects start > end with HTTP 400. Never mutates state.
  • by_outcome sub-dict on every rollup. Each _rollup result (overall, per-category, per time bucket, per category-time) now carries { WIN: {n, total_r, total_pnl}, LOSS: {...}, EOD: {...} }. Keyed by the active result_key — raw or effective — so the same rollup produces the right outcome split under either mode. This is the primitive the UI uses to respect the W/L/EOD chip filter in Cumulative/Range headlines where per-pick rows aren't available.
Fixed
  • Total R and Net $ / share showed "—" in Cumulative mode. renderOverall() was computing headline totals by filtering a picks list and summing r_realized and pnl_dollar. In Single day mode those fields are real numbers on every pick. Sector Leadership Check v3.3

Sector Rotation tells us which sector is hot. Sector Leadership tells us whether this specific ticker is leading, following, or lagging its sector. The two complement each other: a stock can be in a hot sector but still be a laggard within it — and those laggards historically underperform even in favorable conditions.

How It Works

At scan time, the scanner compares three intraday % moves: the ticker's move since the open, its sector ETF's move, and SPY's move. The relative ordering produces a label:

LEADER (ticker% > sector% > SPY%) → ADMIT
SOLO MOVER (ticker > SPY, sector < SPY) → ADMIT
FOLLOWER (ticker > SPY, ≤ sector) → DROP
LAGGARD (ticker weaker than its sector) → DROP
UNKNOWN (sector ETF data unavailable) → DROP

Example: NVDA +2.1%  |  SMH +1.4%  |  SPY +0.5% → LEADER (admit)
Example: AMD +0.4%  |  SMH +1.4%  |  SPY +0.5% → LAGGARD (drop)

As of v3.3.2 this is a hard filter — only LEADER and SOLO MOVER candidates survive to the composite-score stage. The earlier +10/+3/−10 score adjustments are no longer added; admitted tickers must clear the raw min-score on their own technical merits. A 20-day backtest showed the score-boost approach was letting marginal-technical leaders through and causing disproportionate bleed during ELEVATED/HIGH VIX regimes.

Classification is skipped in the first 15 minutes after the open (before 9:45 AM) because early-session % moves are noisy and misleading — during this window all tickers come back UNKNOWN and are dropped.

Operators can override via config.LEADER_FILTER_MODE: "strict" (LEADER only), "moderate" (default, LEADER + SOLO), "permissive" (adds FOLLOWER), "score" (legacy boost-not-gate behaviour).

All 137 scanner tickers are now sector-mapped (v3.3.2), so "UNKNOWN" should only appear in the opening-15-min window or when yfinance is unavailable.

Market Regime Filter (VIX) v3.3

Momentum strategies perform very differently in calm markets (VIX < 15) vs. stressed markets (VIX > 25). On high-VIX days, breakouts fail more often, reversals are sharper, and slippage is worse. Rather than run the same aggressive scan in every environment, v3.3 reads ^VIX at each scan tick and tightens the bar dynamically.

Intraday VIX pop > 10% → bump regime up one band (even from a low base).

What Gets Adjusted

Two things change by regime: (1) the minimum composite score a candidate must clear to generate a signal, and (2) the position-size recommendation. Stop distances, targets, and all indicator calculations are unchanged — it's purely a discipline / sizing overlay.

The scanner falls back to NORMAL if the VIX fetch fails, so a broken data feed never stops scanning.

The regime banner at the top of the dashboard shows the current VIX level, its change vs. the prior close, the regime label, and the effective min-score & size multiplier being used for the current scan.

Earnings Calendar Indicator v3.3

Earnings announcements create discontinuous price jumps that ruin intraday stops — a stock can gap 10% through your stop overnight regardless of how strong the setup looked at 3:59 PM. v3.3 consults a locally-cached earnings calendar and both flags and, in the most dangerous case, blocks.

Data Source & Cache

At 8:00 AM ET each trading day, the scanner pulls the next 14 days of confirmed earnings dates from Financial Modeling Prep. Any ticker in the universe without an FMP hit falls back to yfinance. The combined result is cached locally (6-hour TTL) so scan ticks don't hit the network for every candidate.

The hard block exists because holding a position into an AMC print is a binary event, not a trade. The soft penalty acknowledges that the market often front-runs earnings with elevated volatility, so a setup that's barely clearing score thresholds shouldn't get priority the day before.

Pre-Market Catalyst Scan NEW

Runs at 8:00 AM and 9:00 AM ET, before the market opens. Checks up to 50 tickers for two signals:

1. Volume Ratio = Today's pre-market volume / Yesterday's full-day volume
  If > 2.0× → boost up to +10 pts (capped at 5× volume)

2. News Sentiment (via Finnhub + VADER)
  If sentiment > 0.3 → boost up to +10 pts

Combined boost capped at +15 pts per ticker.
Negative sentiment = no penalty, just no boost.

Critical design principle: This scan is purely additive. It boosts scores for pre-market movers but never prevents a stock from signaling if it only starts moving after the 9:30 AM open. A stock with no pre-market activity can still score 90+ on technicals alone.

News Sentiment Scoring (25% of score)

Headlines are scored using VADER (Valence Aware Dictionary for Sentiment Reasoning), enhanced with a finance-specific lexicon. Words like "bullish", "beat", "upgrade" get strong positive scores; "crash", "downgrade", "miss" get negative scores.

Recency Decay

Recent news matters more. A headline from 10 minutes ago gets full weight; one from 2 hours ago gets half weight.

Weighted Sentiment = Raw Score × Recency Weight

< 15 min old → 1.0× weight
15–30 min → 0.9×
30–60 min → 0.7×
1–2 hours → 0.5×
> 2 hours → 0.3×

First-Order vs. Second-Order Impact

First-order: Direct effect on the company in the headline. Apple beats earnings → Apple stock moves. This is scored directly from the headline.

Second-order: Spillover to related companies. Apple beats earnings → semiconductor suppliers (AVGO, QCOM) benefit from implied demand strength. The scanner tracks sector peer mappings and can flag when a peer has strong recent sentiment.

Data-Driven Filters NEW

These filters were derived from a 3-day backtest (159 trades, April 15–17, 2026). They remove statistically confirmed losing patterns:

Lunch Dead Zone (11:30 AM – 12:00 PM ET)

Scans scheduled at 11:30 and 12:00 are skipped entirely. In 3 days of data, lunch-hour batches produced 0 wins in 21 decided trades. The probability of this happening by chance (given the overall ~30% base win rate) is p < 0.001. Lunch-hour volume dries up and price moves during this window are unreliable.

Re-Entry Suppression

Each ticker can only generate one signal per day (its first appearance). Subsequent scans may detect the same ticker again, but these re-entries are logged and filtered out. In 3 days of data, re-entries produced 1 win in 32 decided trades (3.1% win rate). The pattern: by the time a stock appears a second time, the move has already happened — you're buying the top of a move that's running out of steam.

RVOL Exhaustion Penalty

When RVOL exceeds 5×, the volume component of the composite score is reduced by 30%. Extremely high volume — while attention-grabbing — often signals a climactic move or blow-off top rather than sustainable momentum. The penalty dampens (not eliminates) the signal.

Impact of Filters (3-day results)

Before filters: 159 trades, ~30% win rate
After filters: 85 trades, ~36% win rate

Dead zone removed: 21 trades (0 wins)
Re-entries removed: 32 trades (1 win)
Best time window: 1:00–3:00 PM ET (~67% win rate)

Pivot Point Resistance Levels

In addition to ATR-based targets, each signal includes the nearest Standard Floor Pivot resistance level above the entry price. This gives a Barchart-style overhead resistance estimate based on the prior day's price action.

Pivot (P) = (High + Low + Close) / 3

R1 = (2 × P) − Low
R2 = P + (High − Low)
R3 = High + 2 × (P − Low)

S1 = (2 × P) − High
S2 = P − (High − Low)
S3 = Low − 2 × (High − P)

The scanner finds the nearest R1, R2, or R3 above the current entry price and displays it as the Resistance Target (purple line on the chart). This helps you gauge how much room a trade has before hitting a natural ceiling.

Post-Market Performance Tracking NEW

Every weekday at 5:00 PM ET, the scanner automatically evaluates all of the day's picks:

For each signal from today:
  1. Fetch Yahoo Finance 5-min bars from signal time → 4:00 PM
  2. Check: did price hit the ATR target? → WIN
  3. Check: did price hit the stop-loss? → LOSS
  4. Neither hit → EOD (end-of-day close used as exit)

Results are appended to scanner_trade_log.xlsx (cumulative).
The Performance page reads from this file on every page load.

Duplicate protection is built in — if a day's data is already in the log, it won't be double-counted. The log tracks: date, batch time, ticker, entry price, target, stop, RVOL, RSI, composite score, result (WIN/LOSS/EOD), exit price, P&L in dollars and percent, and whether the signal was a first appearance or re-entry.

Known Limitations

Data source failover: FMP is the primary data source for real-time quotes and 5-minute candles. If the FMP API is unreachable or returns errors, the scanner automatically falls back to yfinance (15-minute delayed data). During market hours, demo data is never shown — if both sources fail, the scan returns empty rather than stale prices.

FMP Starter plan limits: The Starter plan does not support batch-quote endpoints, so quotes are fetched one ticker at a time with rate limiting (4 calls/sec). A full scan of ~130 tickers takes ~30 seconds for quotes alone.

Filter sample size: Dead zone and re-entry filters are based on 3 days of data (159 trades). While statistically significant (p < 0.001 for dead zone), continued monitoring is important. The post-market analysis pipeline will accumulate more data to validate or refine these rules over time.

Sector rotation latency: Sector detection runs hourly (at :30 past each hour). A sudden sector rotation that happens at :45 won't be reflected until the next check at :30. The boost persists until the next detection run.

Pre-market data quality: Pre-market volume data from yfinance can be spotty for smaller-cap names. The scan limits to 50 tickers and uses the prepost=True flag, but coverage may vary.

History persistence: Daily finds are stored as JSON files on disk. On Railway (ephemeral filesystem), history is lost on redeploy. The trade log (scanner_trade_log.xlsx) is committed to the repo to survive redeployments.

News coverage: Finnhub free tier covers major headlines but may miss niche or breaking news. Sentiment scoring is keyword-based (VADER), not deep-learning. It catches obvious sentiment but may miss sarcasm or complex narratives.

Not financial advice: This scanner is an educational tool. Past technical patterns don't guarantee future results. Always do your own research and consider consulting a financial advisor.
