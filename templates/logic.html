v3.5.4 Released 2026-04-21 Current
"GitHub Auto-Backup" — daily belt-and-suspenders snapshot of the persistent data files to a dedicated data-backups branch of this repo, so a wiped volume is always recoverable.
Why: v3.5.3 moved the live data onto a Railway volume, which survives ordinary redeploys. But a volume deletion, accidental unmount, or a new environment still starts from an empty seed. A second copy of performance_log.json and the day's finds, pushed once a day to a GitHub branch that does not trigger the Railway deploy, is cheap insurance: restoration is a 2-line copy, and the branch is browsable in the GitHub UI.
Added
  • data_backup.py module — pure-httpx GitHub Contents API client. Writes each file to data/<basename> on the data-backups branch, creates the branch from main if missing, and skips the PUT when the existing content is byte-identical (so repeated EOD runs don't churn commits). Returns a per-file status map: created, updated, unchanged, missing, or error.
  • Step 7 in daily_analysis.py — right after the EOD performance log is written, the analyzer calls backup_data_files([performance_log.json, YYYY-MM-DD.json]). Wrapped in try/except so any backup failure is logged but never fails the EOD job.
  • Env-gated operation. GITHUB_BACKUP_TOKEN (fine-grained PAT, Contents: read/write on this repo) and GITHUB_BACKUP_REPO (e.g. raveeshsingh/Trader-v3) are required. Missing either → the module logs a one-line no-op and returns cleanly. Optional GITHUB_BACKUP_BRANCH defaults to data-backups.
Changed
  • Backup target is a separate branch, not main. Pushing JSON to main would retrigger a Railway redeploy on every EOD cycle — a feedback loop. The dedicated data-backups branch lives outside the production deploy path.
Fixed
  • Single point of failure on the volume. Before this release, the only copy of the running performance log was the container's volume. Any volume-level accident was unrecoverable. Now every EOD cycle leaves a second copy on GitHub.
Restoring from backup: on the data-backups branch, each file lives at data/<filename>. Download the raw JSON from GitHub, drop it into data/ on the volume (or redeploy with it in data_seed/), and restart the app. The seed helper from v3.5.3 will handle the first-boot copy.
v3.5.3 Released 2026-04-21 Shipped
"Persistent Data Volume" — performance history and daily finds now survive Railway redeploys. Mount a volume at /app/data and the app seeds itself on first boot from data_seed/.
Why: Railway's container filesystem is ephemeral. Every push wiped data/performance_log.json (performance page) and data/2026-MM-DD.json (daily finds / history page) back to whatever was in git. Since data/ is gitignored, that meant both pages reverted to stale/empty state on every deploy. A mounted volume keeps live writes on durable storage; a tiny seed helper keeps the bootstrap dataset discoverable in git.
Added
  • data_seed/ directory (tracked in git) holds pristine bootstrap copies of any JSON we want available on a fresh volume. Currently contains the Apr 20 performance_log.json.
  • _seed_data_volume() startup helper in app.py runs before the scheduler starts. For each file in data_seed/, if the corresponding file does not exist under data/, it's copied in. Idempotent — on a populated volume every file is skipped, and the function logs the per-file counts.
Changed
  • Deployment model. Railway service now has a volume mounted at /app/data. All writes under that path (performance_log.json, daily YYYY-MM-DD.json files, backtest caches, theme-scan snapshots) persist across redeploys.
Fixed
  • History page vanishing on every push. The Home/History dashboards read data/YYYY-MM-DD.json for that day's finds. Before the volume, every Railway redeploy truncated that file back to whatever was in git (usually empty), so the history panel reset to zero on every push. Now the file lives on the volume and simply accumulates.
  • Performance page losing cumulative days. Same root cause — performance_log.json was being reset on redeploy. The EOD job's upsert_day() pattern was already idempotent across days, but the container's writes never survived to accumulate.
v3.5.2 Released 2026-04-20 Shipped
"Email notifier" — strong-signal scans now push an HTML email to your inbox with full entry / stop / target / R:R details, tier pills, and a dashboard link. Weak-only and empty scans stay silent.
Why: Intraday is faster than the dashboard refresh loop. Pulling up the site on every 30-minute scheduler tick is both tedious and prone to missing a strong setup when you're away from the screen. An email push is effectively free (Gmail SMTP + app password) and fits the existing "manual execution at Fidelity" workflow — you still decide whether to enter; the scanner just tells you when there is something worth looking at.
Added
  • New module notifier.py — pure stdlib (smtplib, email.mime, ssl, threading), no new dependencies. Sends via Gmail SMTP on port 587 with STARTTLS. All SMTP work runs on a daemon thread so the scheduler is never blocked waiting for mail.
  • Hook in app.py::scheduled_scan() immediately after add_signals_to_daily(). Fully wrapped in try/except — a mail failure logs an error but never breaks a scan cycle.
  • HTML email layout mirrors the dashboard's dark theme: regime banner at top (color-coded by VIX tier), a compact table of strong signals with tier-colored tickers, and a blue "View dashboard" CTA.
  • Plain-text fallback auto-attached so clients without HTML render (and any email-to-SMS bridges) still see the signals cleanly.
  • .env.example documenting the five new environment variables (see below).
Configuration
  • GMAIL_USER — the sending Gmail address. Required for notifications to fire.
  • GMAIL_APP_PASSWORD — a 16-character Gmail app password (not your account password). Required. Generate at myaccount.google.com/apppasswords. 2FA must be enabled on the account.
  • NOTIFY_EMAIL — comma-separated recipient list. Defaults to GMAIL_USER if unset. You can send to yourself; no need for a separate account.
  • NOTIFY_ENABLED — kill switch. Set to false to silence the notifier without removing creds. Default true.
  • NOTIFY_MIN_SIGNALS — minimum strong signal count to trigger a send. Default 1. Raise to 2 or 3 if you find the volume too high.
  • DASHBOARD_URL — base URL of the Railway deployment, used to link "View dashboard" in the email.
Unchanged (important)
  • Scanner scoring and filter logic are untouched. v3.5.2 is a pure addition — no change to thresholds, tier logic, weak-signal floor, or the performance categorization.
  • Silent when no strong signals fire. Weak-only (40–59) and empty scans do not send email. With the scheduler running ~17 scans/day (15 market-hours ticks + 2 pre-market), expect 3–8 emails on a typical day.
  • Never blocks. Mail send is on a background thread; SMTP errors are logged, not raised.
Operational notes: The first email after deployment confirms Railway's outbound SMTP is open (Gmail on 587 is allowed by default). If no email arrives on a scan that should have fired, check Railway logs for notifier: lines — the module logs every decision (disabled / below threshold / missing creds / sent). App passwords expire if 2FA is disabled on the Gmail account — keep 2FA on. The app password itself does not expire automatically.
v3.5 Released 2026-04-20
"Performance Categories" — /performance rebuilt around a 4-category split: score ≥60 + Leader vs. score ≥60 + non-Leader vs. low-score vs. unclassified. Data store reset to start fresh from today.
Why: The old performance page rolled every trade into one bucket, which hid the fact that high-score Leaders and high-score non-Leaders behave very differently. Separating them makes the P&L narrative legible and lets us tune the scanner around the bucket that's actually making money.
Added
  • New categorization schema stored per-entry at ingest time:
    • A — High-Score Leader: score ≥ 60 AND leadership_label = LEADER
    • B — High-Score Non-Leader: score ≥ 60 AND leadership_label in {FOLLOWER, LAGGARD, SOLO_MOVER}
    • C — Low-Score: score < 60 AND classified leadership label
    • D — Unclassified: no leadership label (early-session picks before sector rotation has settled)
  • New read-side module performance_engine.py (pure stdlib, no new deps) with assign_category, normalize_entry, upsert_day, and build_view. The view dict feeds the template directly.
  • New store data/performance_log.json (versioned wrapper, seeded from today). Day-level upserts are idempotent — re-running the EOD job for the same date replaces that day's rows rather than duplicating.
  • New JSON API GET /api/performance?date=YYYY-MM-DD — returns the same dict the template receives.
  • Dashboard rewrite: single-page dark-theme layout with a mode toggle (Cumulative / Single day), category chips, result chips, four category panels with mini time-bucket tables, a full category × time-of-day cross-tab, and a sortable picks table (single-day only).
  • Diagnostics banner surfaces the two known scanner data-quality bugs: stale ticker remap SQ → XYZ (Block, Inc. rebranded Jan 2025) and post-close picks that should be gated at 15:55 ET in the scheduler.
Changed
  • EOD job (daily_analysis.py::analyze_day) now filters picks to [09:30, 16:00) ET (pre-open and post-close dropped), remaps SQ → XYZ before Yahoo fetch, enriches each resolved pick with its leadership label, and writes to performance_log.json via upsert_day.
  • /performance route now reads from performance_log.json via performance_engine.get_view() instead of scanner_trade_log.xlsx. The Excel log is still written for archival continuity but is no longer the dashboard source.
Added (v3.5.1)
  • Result-mode toggle in the controls bar — Raw vs. P&L-adjusted. An EOD pick is one we held into the close because neither the target nor the stop was hit. Under P&L-adjusted, those picks reclassify by the sign of the close: EOD with positive P&L counts as a WIN, EOD with negative P&L counts as a LOSS, and an exact-zero close stays EOD. The toggle is retroactive across the whole log and is the default view (because the raw EOD bucket is structurally undecided — most close-only exits ended directionally and should be credited).
    • normalize_entry() now stamps an effective_result field alongside result.
    • _rollup() takes a result_key param so the same machinery produces both rollups; build_view() ships {raw, effective} sub-dicts for every cumulative and single-day slice.
    • P&L and R totals are identical across modes — only WIN/LOSS/EOD counts and the derived win rate change.
    • In the picks table, an upgraded badge is annotated with a faint * and a tooltip ("Reclassified from EOD by P&L sign") so reclassifications are visible at a glance.
    First day's effect (2026-04-20, 65 entries): WR moves from 19.5% → 41.5%, Cat A from 20% → 60%, Cat B from 60% → 67% — 24 of 24 EOD picks reclassified (19 WIN, 5 LOSS).
Fixed
  • Category D bloat (upstream fix). 29 of today's 65 picks landed in D (Unclassified) because the scanner fired during windows when _last_rotation_snapshot was empty or stale. Two changes in app.py:
    • Startup warm-up: sector_rotation_job() runs once at FastAPI startup (if market hours), so the first scan after a restart already has context — no more Category D spike on mid-session deploys.
    • Self-heal guard: new _ensure_fresh_sector_snapshot() runs inline at the top of every scheduled_scan. If the snapshot is missing or older than 20 minutes, it re-runs sector_rotation_job synchronously before the scan. Race between the two APScheduler jobs at :30 can no longer leave picks label-less.
    Net effect: Categories A/B/C should absorb the picks that used to land in D. Measure on the next trading day.
Removed
  • Legacy aggregate-only dashboard. The old page's "pooled trades, one WR number" view is gone — every roll-up on the new page is scoped to a specific category, so a global headline number that blends A/B/C/D is intentionally not displayed.
  • Pre-v3.5 performance history is not migrated. The new store starts fresh from 2026-04-20. Historical trades remain in the Excel log for audit, but the dashboard only tracks the new categorized era.
First day's split (2026-04-20, 65 entries): A=10, B=9, C=17, D=29. The size of bucket D is an early-session artefact — the scanner fires before the sector-rotation job completes, so those picks never get a leadership label. The fix lives upstream in the scanner, not here. The Unclassified bucket is exposed on the page so we can watch it shrink as the scheduler tightens.

Added

v3.4.2 Released 2026-04-20 Shipped
"Unblock the dashboard" — two scanner bugs fixed, leader gate softened to tiered display, weak signal floor added at score 40.
Fixed
  • Partial-bar RVOL bug. calculate_rvol was reading df["Volume"].iloc[-1] — which on yfinance's live feed is the currently accumulating 5-min bar. Mid-bar that's ~20 seconds of volume being compared against full historical 5-min bars, collapsing RVOL below 0.5 for nearly every ticker and killing almost all signals. The fix detects partial bars via now < bar_end and falls back to iloc[-2] (the most recent complete bar) when the last bar hasn't closed yet. Validation on 2026-04-20 live data: RVOL distribution went from "127 of 135 below 0.5" to a realistic spread with 8 above 2.0 and 25 in 1.0–1.33.
  • Timezone mismatch in intraday % change. _calc_ticker_intraday_pct compared a tz-aware df.index.normalize() against a tz-naive pd.Timestamp(today).normalize(), silently raising and falling back to None for every ticker — which is why the UI showed "0.00%" on every row all day. Fix rewrites the comparison as per-element .date() on tz-aware timestamps, which is tz-safe. After fix, 23 of 135 tickers on the live scan had non-zero ticker_pct values.
Changed
  • Leader gate is now a display tier, not a hard filter. The v3.3.2 LEADER+SOLO-only hard-filter was blocking legitimate signals on days when the sector-leader computation disagreed with the scanner's technicals (most commonly in rotation / chop regimes). v3.4.2 introduces LEADER_FILTER_MODE = "display" — all candidates above the composite threshold are emitted; the leader tier is shown as a visual pill:
    • LEADER / SOLO → Primary picks (top of list)
    • FOLLOWER → Secondary (sector moving but ticker isn't the leader — still tradeable, clearly labeled)
    • LAGGARD / UNKNOWN → Unclassified (shown but visually de-emphasized)
    The old "moderate" and "aggressive" modes remain available in config.py for backtest comparability; backtest.py Filters logic is untouched so historical comparisons still work.
  • Weak-signal floor added at score 40. MIN_COMPOSITE_SCORE stays at 60 as the strong threshold; new WEAK_SIGNAL_FLOOR = 40 surfaces candidates in the 40–59 band as a clearly-labeled "weak signal" section below the strong list, with a divider and a "not recommended for entry, but the setup is forming" note. Intent: when the strong list is empty, give the user situational awareness of what the scanner is tracking rather than a blank page.
  • MAX_SIGNALS_PER_SCAN raised from 10 → 20 to accommodate the combined strong+weak display.
Added
  • New CSS tier pills on index.html, today.html, history.html: .badge.follower (amber), .badge.unclassified (muted), .badge.weak-signal (dashed border). Strong signals and weak signals are visually separated by a .signals-section-divider with a caption.
  • New signal fields on every emitted signal: leader_tier ("leader" | "solo" | "follower" | "laggard" | "unknown") and signal_strength ("strong" | "weak"). Both are persisted to the Excel trade log for downstream analysis.
Unchanged (important)
  • Composite scoring weights unchanged — still 65% technical + 25% sentiment + 10% volume. The v3.4.2 changes are entirely in the emission and display stages.
  • Backtest default still FLAT (v3.4.1 flipped it to MAE_EXIT briefly during research; FLAT with the 60-min manual MAE rule remains the recommended live workflow).
  • 60-minute MAE manual rule from v3.4.1 is unchanged and remains the primary exit discipline.
Why it matters: At 10:30 AM ET on 2026-04-20 the dashboard was showing zero signals despite a normal-VIX tape with plenty of stocks up 0.6–1.2% intraday. A per-stage diagnostic traced the drop-off to the RVOL filter (127 of 135 tickers below 1.33), and the RVOL distribution itself looked impossibly skewed — which surfaced the partial-bar bug. The leader-gate softening + weak-floor are the UX-layer response to "when the strong list is empty, what should the user see?" — it's better to show clearly-labeled weaker setups than a blank dashboard, because situational awareness matters even when no trade is warranted.
v3.4.1 Released 2026-04-18
"60-minute MAE check" — backtest-validated optional manual exit rule. No live scanner changes.
Added
  • New optional manual trading rule — the 60-minute MAE check. When running a trade on Fidelity, if both of the following are true at the 60-minute mark after entry, close the position flat:
    1. The stock at some point dropped more than 0.75R below your entry (e.g., entry $100, stop $96 → R = $4 → threshold is $97 or lower), and
    2. The stock is still below your entry price at the 60-min candle close.
    Over 20 trading days this rule would have saved $2,611 across 10 manual exits, with only 2 false-positives (trades closed that would have turned positive).
  • MAE_EXIT simulator mode in backtest.py — codifies the rule above with constants MAE_EXIT_BAR = 12 (bars), MAE_EXIT_MAE_R = −0.75, MAE_EXIT_MAX_CUR_R = 0.0. Stop and target checks still take priority at every bar.
  • Bar-8 and bar-12 MFE/MAE/R snapshots on every backtest trade for future rule research.
  • Two additional exit variants simulated for comparison: MFE_KILL (close at bar 8 if MFE < 0.3R — rejected, see below) and TIGHT_ABANDON (bar-12 chop + MAE < −0.5R — worse than MAE_EXIT).
Unchanged (important)
  • Live scanner: untouched. scanner.py and config.py have zero v3.4 / v3.4.1 references. The 60-minute MAE rule is a manual rule the user applies at Fidelity — there is no automated closing.
  • Backtest default exit remains FLAT (target / stop / EOD). MAE_EXIT is a simulator-only parallel track.
How the rule was found (honest methodology):
  1. v3.4 hypothesis: "Close any trade whose MFE hasn't reached +0.3R by 40 min — should save ~$35–40K over 20 days" (based on the 48 trades that never exceed +0.5R MFE lifetime, which bleed $46K).
  2. Implementation: added MFE_KILL as a parallel stream in the simulator, re-ran the 20-day backtest.
  3. Reality check: MFE_KILL at 0.3R cost $6,149, not saved $35K. The rule closed 12 real winners that happened to be sitting below +0.3R at bar 8. The "48 never-profitable" trades don't identify themselves reliably that early — many poke above +0.3R then fade.
  4. Parameter sweep: tested 8 MFE thresholds at bar 8 and 7 gate-combinations at bar 12. Best rule was MAE < −0.75R AND current R < 0 at bar 12 — fires 11 times, 2 false-positives, saves $2,379.
  5. Codified as MAE_EXIT and re-ran the simulation. Final: +$2,611 P&L, $2,289 less drawdown, 10 fires, 2 false-positives.
Why MAE beats MFE as the signal: MFE answers "has the trade ever looked good?" — a noisy, high-false-positive question. MAE answers "has the trade actually gone against me?" — much more selective. Combined with a "not recovering" gate (current R < 0), it catches genuine losers-in-progress without culling winners that briefly pulled back.

Expected to remain positive because it rides a genuine causal signal (MAE is a leading indicator of stops), but the magnitude may shrink.
v3.4 Released 2026-04-18
"Exit-strategy laboratory" — win redefined, MFE/MAE tracking, four parallel exit simulations.
Changed
  • Win definition redefined. A trade is now a "win" if final P&L is positive, not only if the +2.5R target is hit. The old TARGET-hit rate is still reported as target_hits for reference. This surfaced that the scanner's real profitable rate was 46% (58/126), not the headline 26.2% — 37 EOD-win trades that banked +$27,541 were being mislabelled as not-wins.
Added
  • MFE / MAE tracking per trade — peak favorable and adverse excursion in R units, plus the bar index at which each peak occurred. Enables all subsequent exit-logic research.
  • TIERED exit simulation — 33% off at +1R (stop → breakeven) / 33% off at +1.75R (trailing stop on prev-bar low) / 33% runs to target or EOD.
  • ABANDON rule simulation ("1.3 rule") — if at bar 12 (60 min) the trade is inside ±0.5R of entry, close flat at the bar close.
  • COMBO simulation — TIERED exits + abandon rule applied only while no tranches have been banked.
  • Aggregate mfe_mae_stats, tiered_summary, abandon_summary, combo_summary blocks on every backtest result.
Why: v3.3.2 cut the scanner's entry noise but left drawdowns sizable ($20K max in the baseline run) and the "real WR" question unresolved. This release was a pure exit-logic research pass to (a) get the true WR on the books and (b) quantify whether tiered exits / time-based abandonment would help.

Findings from the 20-day backtest:
  • Tiered exits halve P&L. WR jumps to 54.2% but total P&L drops $14.7K because tiered caps every runner. Of 24 trades that hit ≥2R MFE, FLAT earned $54,836 vs TIERED $33,818 — a $21K giveaway.
  • The abandon rule (|R| < 0.5 at bar 12) killed slow winners. BKNG on 2026-04-06 hit target at $+1,847 on bar 61 with MFE +2.50R; COMBO abandoned it at bar 12 for $−243. BRK-B on 2026-04-09: FLAT $+2,395, COMBO $+153.
  • The real leak is the 48 never-profitable trades (33% of volume, bleed $46K). They look identical to the profitable cohort at entry (same composite, tech, RVOL, leadership, regime) but differ in what happens in the first ~10 minutes: avg bars held 11.6 vs 38.3, avg MFE +0.18R vs +1.44R.
Conclusion: don't default to tiered or abandon — they both hurt P&L. The exit-rule work is worth continuing, but with tighter, MAE-aware gates. See v3.4.1 above for the follow-up.

Added
  • Full sector mapping — all 137 scanner tickers now have a sector assignment (up from 80/137). Signal Generation Pipeline

1

Pre-Market Catalyst Scan NEW

At 8:00 and 9:00 AM ET, the scanner checks up to 50 tickers for unusual pre-market volume (>2× prior day) and bullish news sentiment. Flagged tickers receive a score boost (up to +15 points) during the main scan, but no ticker is ever filtered out by this step — stocks that only start moving after the open are still scanned normally.

2

Sector Rotation Detection NEW

Starting at 9:30 AM and every hour after, 15 sector ETFs (XLK, SMH, IGV, CLOU, XLF, XLV, XLE, XLY, XLI, XLB, XLU, XLRE, XLC, XLP, BITO) are compared against SPY. The top 3 sectors by relative strength are identified, and their high-beta component stocks receive a +8 point score boost during the main scan.

3

Volume Filter (Gate)

Fetch 5-minute candles for ~130 stocks (100 liquid S&P 500 large-caps + ~30 high-beta mid/small-caps) via FMP real-time API (with yfinance fallback). Calculate time-of-day-adjusted Relative Volume (RVOL). Stocks with RVOL < 1.33× are immediately discarded — insufficient institutional interest.

4

Technical Scoring (65% weight)

Score each surviving stock on 5 technical dimensions: EMA alignment (25 pts), RSI momentum zone (20 pts), MACD trend (20 pts), VWAP position with crossover detection (20 pts), and Bollinger Band position (20 pts). Total is normalized to 0–100.

5

News Sentiment Scoring (25% weight)

Pull recent news headlines from Finnhub. Score each headline with VADER (finance-tuned NLP). Apply recency decay: news <15 min old gets full weight, >1 hour old gets 30% weight. Aggregate into −1.0 to +1.0 sentiment score.

6

Composite Score & Legacy Boosts

Base composite = (Technical × 0.65) + (Sentiment × 0.25) + (Volume × 0.10). Then apply: sector rotation boost (+8 pts if in top 3 sectors), pre-market catalyst boost (up to +15 pts), and RVOL >5× exhaustion penalty (−30%).

7

Context Gates — Leadership / Regime / Earnings v3.3.2

Sector Leadership (HARD GATE in v3.3.2): Compare the ticker's intraday % move against its sector ETF and SPY, classify as LEADER (ticker > sector > SPY), SOLO_MOVER (ticker > SPY but sector lagging SPY), FOLLOWER (ticker > SPY but lags its sector) or LAGGARD (ticker < sector). In LEADER_FILTER_MODE="moderate" (the default) only LEADER + SOLO_MOVER pass; FOLLOWER, LAGGARD and UNKNOWN are dropped entirely — no score penalty, no signal. See the v3.3.2 release notes below for the backtest rationale and alternate modes (strict, permissive, legacy score). Market Regime (VIX): effective minimum score is bumped by regime — CALM/NORMAL 60, ELEVATED 65, HIGH 70, EXTREME 80. Position-size recommendation scales 100 → 25 % across bands. Earnings: hard-block entries after 2:00 PM ET on a post-close print day; soft −5 pts if earnings are tomorrow. Pills on the dashboard surface leadership, regime and earnings status.

8

Dead Zone & Re-Entry Filters

Dead Zone: Scans at 11:30 AM and 12:00 PM ET are skipped entirely — across 3 days of data, lunch-hour batches produced 0 wins in 21 decided trades (p < 0.001). Re-Entry Suppression: Each ticker can only signal once per day. Re-entries (same ticker appearing in a later scan) are logged but filtered out — they produced only 1 win in 32 decided trades (3.1% win rate). Remaining stocks scoring below the effective min-score are dropped; top 10 are kept.

9

Trade Levels & Signal Output

For each signal: Entry = current price. Stop-loss = Entry − (ATR × 2.0). ATR Target = Entry + (stop distance × 2.5). Resistance Target = nearest pivot-point resistance (R1/R2/R3) above entry, calculated from prior day's H/L/C using the Standard Floor Pivot formula. Position size = 1% account risk / stop distance, scaled by the regime size multiplier. Hard close by 3:55 PM ET.

Relative Volume (RVOL)

RVOL compares current volume to the average volume at the same time of day over the past 20 sessions. This matters because volume naturally spikes at open (9:30–10:00 AM) and close (3:30–4:00 PM), and dips at midday. A simple daily average would show false positives in the morning and false negatives at noon.

RVOL(now) = Volume(now) / Avg Volume(same time of day, past 20 days)

Thresholds:
  < 1.0x → Below average (SKIP)
  1.33x → Minimum for entry consideration
  2.0x+ → Strong institutional interest
  3.0x+ → Catalyst event likely
  5.0x+ → EXHAUSTION PENALTY: volume score reduced 30%

The RVOL >5× penalty is new in v3.2 — extremely high volume can signal exhaustion or blow-off tops rather than opportunity. The volume score is still positive, just dampened.

Sector Leadership Check v3.3

Sector Rotation tells us which sector is hot. Sector Leadership tells us whether this specific ticker is leading, following, or lagging its sector. The two complement each other: a stock can be in a hot sector but still be a laggard within it — and those laggards historically underperform even in favorable conditions.

How It Works

At scan time, the scanner compares three intraday % moves: the ticker's move since the open, its sector ETF's move, and SPY's move. The relative ordering produces a label:

LEADER (ticker% > sector% > SPY%) → ADMIT
SOLO MOVER (ticker > SPY, sector < SPY) → ADMIT
FOLLOWER (ticker > SPY, ≤ sector) → DROP
LAGGARD (ticker weaker than its sector) → DROP
UNKNOWN (sector ETF data unavailable) → DROP

Example: NVDA +2.1%  |  SMH +1.4%  |  SPY +0.5% → LEADER (admit)
Example: AMD +0.4%  |  SMH +1.4%  |  SPY +0.5% → LAGGARD (drop)

As of v3.3.2 this is a hard filter — only LEADER and SOLO MOVER candidates survive to the composite-score stage. The earlier +10/+3/−10 score adjustments are no longer added; admitted tickers must clear the raw min-score on their own technical merits. A 20-day backtest showed the score-boost approach was letting marginal-technical leaders through and causing disproportionate bleed during ELEVATED/HIGH VIX regimes.

Classification is skipped in the first 15 minutes after the open (before 9:45 AM) because early-session % moves are noisy and misleading — during this window all tickers come back UNKNOWN and are dropped.

Operators can override via config.LEADER_FILTER_MODE: "strict" (LEADER only), "moderate" (default, LEADER + SOLO), "permissive" (adds FOLLOWER), "score" (legacy boost-not-gate behaviour).

All 137 scanner tickers are now sector-mapped (v3.3.2), so "UNKNOWN" should only appear in the opening-15-min window or when yfinance is unavailable.

Market Regime Filter (VIX) v3.3

Momentum strategies perform very differently in calm markets (VIX < 15) vs. stressed markets (VIX > 25). On high-VIX days, breakouts fail more often, reversals are sharper, and slippage is worse. Rather than run the same aggressive scan in every environment, v3.3 reads ^VIX at each scan tick and tightens the bar dynamically.

What Gets Adjusted

Two things change by regime: (1) the minimum composite score a candidate must clear to generate a signal, and (2) the position-size recommendation. Stop distances, targets, and all indicator calculations are unchanged — it's purely a discipline / sizing overlay.

The scanner falls back to NORMAL if the VIX fetch fails, so a broken data feed never stops scanning.

The regime banner at the top of the dashboard shows the current VIX level, its change vs. the prior close, the regime label, and the effective min-score & size multiplier being used for the current scan.

Earnings Calendar Indicator v3.3

Earnings announcements create discontinuous price jumps that ruin intraday stops — a stock can gap 10% through your stop overnight regardless of how strong the setup looked at 3:59 PM. v3.3 consults a locally-cached earnings calendar and both flags and, in the most dangerous case, blocks.

At 8:00 AM ET each trading day, the scanner pulls the next 14 days of confirmed earnings dates from Financial Modeling Prep. Any ticker in the universe without an FMP hit falls back to yfinance. The combined result is cached locally (6-hour TTL) so scan ticks don't hit the network for every candidate.

Pre-Market Catalyst Scan NEW

Runs at 8:00 AM and 9:00 AM ET, before the market opens. Checks up to 50 tickers for two signals:

1. Volume Ratio = Today's pre-market volume / Yesterday's full-day volume
  If > 2.0× → boost up to +10 pts (capped at 5× volume)

2. News Sentiment (via Finnhub + VADER)
  If sentiment > 0.3 → boost up to +10 pts

Combined boost capped at +15 pts per ticker.
Negative sentiment = no penalty, just no boost.

Critical design principle: This scan is purely additive. It boosts scores for pre-market movers but never prevents a stock from signaling if it only starts moving after the 9:30 AM open. A stock with no pre-market activity can still score 90+ on technicals alone.

News Sentiment Scoring (25% of score)

Headlines are scored using VADER (Valence Aware Dictionary for Sentiment Reasoning), enhanced with a finance-specific lexicon. Words like "bullish", "beat", "upgrade" get strong positive scores; "crash", "downgrade", "miss" get negative scores.

Recency Decay

Recent news matters more. A headline from 10 minutes ago gets full weight; one from 2 hours ago gets half weight.

Weighted Sentiment = Raw Score × Recency Weight

< 15 min old → 1.0× weight
15–30 min → 0.9×
30–60 min → 0.7×
1–2 hours → 0.5×
> 2 hours → 0.3×

First-Order vs. Second-Order Impact

First-order: Direct effect on the company in the headline. Apple beats earnings → Apple stock moves. This is scored directly from the headline.

Second-order: Spillover to related companies. Apple beats earnings → semiconductor suppliers (AVGO, QCOM) benefit from implied demand strength. The scanner tracks sector peer mappings and can flag when a peer has strong recent sentiment.

Data-Driven Filters NEW

These filters were derived from a 3-day backtest (159 trades, April 15–17, 2026). They remove statistically confirmed losing patterns:

Lunch Dead Zone (11:30 AM – 12:00 PM ET)

Scans scheduled at 11:30 and 12:00 are skipped entirely. In 3 days of data, lunch-hour batches produced 0 wins in 21 decided trades. The probability of this happening by chance (given the overall ~30% base win rate) is p < 0.001. Lunch-hour volume dries up and price moves during this window are unreliable.

Re-Entry Suppression

Each ticker can only generate one signal per day (its first appearance). Subsequent scans may detect the same ticker again, but these re-entries are logged and filtered out. In 3 days of data, re-entries produced 1 win in 32 decided trades (3.1% win rate). The pattern: by the time a stock appears a second time, the move has already happened — you're buying the top of a move that's running out of steam.

RVOL Exhaustion Penalty

When RVOL exceeds 5×, the volume component of the composite score is reduced by 30%. Extremely high volume — while attention-grabbing — often signals a climactic move or blow-off top rather than sustainable momentum. The penalty dampens (not eliminates) the signal.

Impact of Filters (3-day results)

Before filters: 159 trades, ~30% win rate
After filters: 85 trades, ~36% win rate

Dead zone removed: 21 trades (0 wins)
Re-entries removed: 32 trades (1 win)
Best time window: 1:00–3:00 PM ET (~67% win rate)

Pivot Point Resistance Levels

In addition to ATR-based targets, each signal includes the nearest Standard Floor Pivot resistance level above the entry price. This gives a Barchart-style overhead resistance estimate based on the prior day's price action.

Pivot (P) = (High + Low + Close) / 3

R1 = (2 × P) − Low
R2 = P + (High − Low)
R3 = High + 2 × (P − Low)

S1 = (2 × P) − High
S2 = P − (High − Low)
S3 = Low − 2 × (High − P)

The scanner finds the nearest R1, R2, or R3 above the current entry price and displays it as the Resistance Target (purple line on the chart). This helps you gauge how much room a trade has before hitting a natural ceiling.

Known Limitations

Data source failover: FMP is the primary data source for real-time quotes and 5-minute candles. If the FMP API is unreachable or returns errors, the scanner automatically falls back to yfinance (15-minute delayed data). During market hours, demo data is never shown — if both sources fail, the scan returns empty rather than stale prices.

FMP Starter plan limits: The Starter plan does not support batch-quote endpoints, so quotes are fetched one ticker at a time with rate limiting (4 calls/sec). A full scan of ~130 tickers takes ~30 seconds for quotes alone.

Filter sample size: Dead zone and re-entry filters are based on 3 days of data (159 trades). While statistically significant (p < 0.001 for dead zone), continued monitoring is important. The post-market analysis pipeline will accumulate more data to validate or refine these rules over time.

Sector rotation latency: Sector detection runs hourly (at :30 past each hour). A sudden sector rotation that happens at :45 won't be reflected until the next check at :30. The boost persists until the next detection run.

Pre-market data quality: Pre-market volume data from yfinance can be spotty for smaller-cap names. The scan limits to 50 tickers and uses the prepost=True flag, but coverage may vary.

History persistence: Daily finds are stored as JSON files on disk. On Railway (ephemeral filesystem), history is lost on redeploy. The trade log (scanner_trade_log.xlsx) is committed to the repo to survive redeployments.

News coverage: Finnhub free tier covers major headlines but may miss niche or breaking news. Sentiment scoring is keyword-based (VADER), not deep-learning. It catches obvious sentiment but may miss sarcasm or complex narratives.
